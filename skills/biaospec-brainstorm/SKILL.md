# biaospec-brainstorm — 访谈式需求澄清

## 核心指令

围绕这个需求的每一个方面持续访谈我，直到我们达成共享理解。把需求当成一棵决策树，沿每个分支走下去，按依赖关系逐个解决——上游决策先于下游。

- 一次只问一个问题，每个问题都给出你的推荐答案和理由
- 能靠探索代码库回答的问题，直接去查，不要问我
- 有多个可选项时优先出选择题（2-4 个选项，推荐项放最前）；给问题标个进度前缀（如 `[访谈 2/N]`）方便我定位

## 流程

1. **探索上下文**：先看相关代码、docs/、近期 commit，把能自己回答的问题消化掉，再开口提问
2. **访谈澄清**：按核心指令逐个解决目的、约束、成功标准；每得到一个回答，向决策树下一层走
3. **提方案（仅在仍有分歧时）**：如果访谈后仍存在多个可行方向，给 2-3 个方案（各带 trade-off）亮明推荐让用户选；如果访谈已把方案收敛成一条，跳过这步，别硬凑平行方案
4. **呈现设计 = 落盘 spec 草稿供评审**：把设计直接写成 `.biaospec/YYYY-MM-DD-<主题>/spec.md`（一个任务一个目录，任务间互不干扰；`YYYY-MM-DD` 用今天的真实日期，不要用记忆里的旧日期；状态="需求澄清中"，模板见下），让用户在文件上评审，而不是先在会话里口述一遍再落盘。简单任务的设计可以短到几句话，不要为小事铺排大文档。写完逐项自审，发现问题直接修：
   - placeholder：有无 TBD / TODO / 含糊需求
   - 自洽：各节之间是否矛盾
   - 范围：是否单次可实施，过大则建议拆成多个 spec
   - 歧义：有无可作两种解读的表述
   - 覆盖面（限复杂任务）：错误处理、测试策略（涉及安全 / 性能时一并）是否都有着落
5. **审阅移交**：请用户审阅 spec 文件；获批后把状态行改为"已批准"，建议加载 `biaospec-plan` 拆实施计划

## 设计门

**用户批准设计之前，不写任何实现代码、不搭任何脚手架。**

## spec 模板

```markdown
# <标题>

> 状态：需求澄清中 | 已批准 | 实施中 | 完成

## 需求（PRD）

### 目标

<一段话：解决什么问题、为什么做>

### 验收标准

- [ ] AC-1 <可测试的判据，尽量写成"当 X 时，系统应 Y"；编号供实施计划回指>
- [ ] AC-2 <…>

### Out of Scope

- <明确不做什么>

## 设计（Spec）

<方案与关键决策，含选型理由和被否掉的备选；简单任务几句话即可。
子系统 / 多组件级任务则往上展开，按需取用下列维度（不是必填清单）：
- 默认写：总体架构（含数据流）、关键类型 / 接口契约
- 数据模型：涉及持久化 / 新数据结构时
- 框架契合与侵入性：在存量工程内展开时——顺现有结构扩展、权衡改动侵入性、别无脑加增量，"扩展现有 A 还是新增并行 B"这类架构分叉抛给用户拍板
- 分期与风险：子系统级——V1 最小范围 vs 明确后置的硬化项、风险与开放问题>

## 实施计划（Plan）

<由 biaospec-plan 追加>
```

## 反模式

- 一条消息里问多个问题
- 提问不带推荐答案
- 能查代码回答的事拿去问用户
- 设计未获批就动手写代码
