# biaospec-plan — 任务分解与执行纪律

## 前置检查

优先复用当前对话里已知的 spec 路径（brainstorm 刚移交的、或用户消息里点出的任务/文件）——多数情况下 brainstorm 和 plan 是同一次对话里前后脚发生，路径本就在上下文里，不需要去翻文件系统。

只有路径不明确时才去 `.biaospec/` 下找，且不要凭"最近改动/最近日期"猜——任务目录的新旧和"该拆哪个"没有必然关系：

1. 只看状态行为"已批准"的 `spec.md`
2. 用户提到了任务 / 主题关键词就按关键词匹配目录名收窄
3. 收窄后剩一个 → 直接用；剩 0 个或 2+ 个 → 列出候选（目录名 + 一句话目标）请用户指认

没有 spec、或设计仍有含糊之处 → 先回 `biaospec-brainstorm` 澄清，不要凭空拆任务。

## 任务分解

写计划时假设执行者是一位不了解本代码库的熟练工程师：

- 先列文件结构：这次改动会创建 / 修改哪些文件，各自职责一句话
- 每个 task 是一个独立可验证的变更单元，标注涉及文件的精确路径
- 每个 task 标注它覆盖 spec 的哪几条验收标准（`对应：AC-x`）
- 不为拆而拆：同一文件的内聚改动算一个 task，别硬切成"写头部/写主体/写尾部"
- 每个 task 写清怎么验证，验证方式随任务类型：
  - 代码任务 → 跑命令看结果（如 `bun test ...`、`bun run typecheck`）
  - 文档 / skill / 配置任务 → 人工走查 + 端到端手测（如 `grep` 命中关键词、肉眼核对结构齐全、跑一遍看产物）
- task 按依赖排序，可并行的标注出来

### No Placeholders 铁律

以下写法是计划缺陷，禁止出现：

- "TBD" / "TODO" / "后续补充"
- "适当处理错误" / "处理边界情况"——不说清是哪些错误、哪些边界
- "同 Task N"——执行时可能乱序读，关键内容要重复写
- 引用任何 task 里都没定义过的类型 / 函数 / 接口名

## 计划落盘

追加到 spec 文件的 `## 实施计划（Plan）` 节：

```markdown
## 实施计划（Plan）

- [ ] **Task 1：<名称>**（代码任务示例）
  - 文件：创建 `path/to/new.ts`；修改 `path/to/existing.ts`；测试 `path/to/new.test.ts`
  - 内容：<做什么；关键接口签名 / 数据结构直接写出来>
  - 对应：AC-1、AC-3
  - 验证：`bun test path/to/new.test.ts` → 全部通过
- [ ] **Task 2：<名称>**（文档 / skill 任务示例，可与 Task 1 并行）
  - 文件：创建 `path/to/doc.md`
  - 内容：<写哪几块；关键字段 / 模板直接写出来>
  - 对应：AC-2
  - 验证：`grep -i "<关键词>" path/to/doc.md` 命中；肉眼核对各块齐全
```

## 计划自审（三项，发现即改）

1. **覆盖率**：按 AC 编号逐条核对，每条验收标准至少被一个 task 的"对应"行引用；有缺口就补 task
2. **placeholder**：按铁律逐条排查
3. **一致性**：后面 task 引用的命名 / 签名与前面 task 定义的一致

自审后请用户审阅计划；获批后把状态行改为"实施中"再动手。

## 执行纪律

- 按依赖顺序执行，完成一个 task 立刻勾掉它的 checkbox，再开下一个
- 每个 task 完成即跑它的"验证"，不攒到最后一起验
- 实施中发现的关键事实（坑、约定、与预期不符的行为）追加到该 task 条目下（`发现：…`），供后续 task 参考——对话会被压缩，文件不会
- 执行中发现 spec 有缺陷或含糊 → 停下回 `biaospec-brainstorm` 澄清并更新 spec，不自作主张偏离
- 全部完成后：逐条核对验收标准并勾选，状态行改为"完成"；如有偏离 spec 之处，在文末补一行说明

## 反模式

- spec 未获批就开拆
- 计划里留 placeholder
- 一口气实现多个 task，不勾选、不验证
- 发现 spec 问题后静默绕过
